# Planflare: architecture

A concise map of the codebase for picking up work with fresh context. See
[`docs/VISION.md`](VISION.md) for the product rationale and [`docs/DESIGN.md`](DESIGN.md) for
the visual system.

## Layout

```
crates/parser/          Rust workspace member, compiled to wasm32-unknown-unknown
  src/model.rs           PlanNode, Engine, ParseError - the one tree shape all engines converge on
  src/postgres.rs         Auto-detects text or FORMAT JSON; normalizes either into PlanNode
  src/mysql.rs            EXPLAIN FORMAT=JSON parser (recursive descent over serde_json::Value)
  src/sqlite.rs           EXPLAIN QUERY PLAN parser (id/parent linkage -> tree)
  src/lib.rs              parse(engine, text) dispatch + #[wasm_bindgen] parse_plan JS entrypoint
  tests/basic.rs          cross-engine integration tests
scripts/build-wasm.sh    cargo build (wasm32) + wasm-bindgen -> web/src/wasm/ (generated, gitignored)
scripts/build-site.sh    runs the Vite production build that emits into site/
site/                    generated, deployable app bundle with relative asset paths
web/                     TypeScript + Vite UI, no framework
  src/plan.ts            PlanNode TS type + selfTimeMs/findHottestNode/isRowEstimateMismatch/
                          findHottestNodePath/pathKey/describeNodeFields
  src/parser.ts          wraps wasm init + parse_plan, normalizes thrown strings into Error
  src/tree-view.ts       recursive DOM renderer; returns a TreeController ({element, focusPath,
                          getCollapsedPaths}) - collapse/expand, hotspot highlight, mismatch
                          badge, per-node path tracking, onToggle/onNodeSelect hooks
  src/inspector.ts       click-to-inspect dialog: every non-null PlanNode field, Escape/
                          backdrop-click to close, focus returns to the row that opened it
  src/storage.ts         localStorage persistence for the last-pasted plan (engine, text,
                          collapsed paths) - guarded against unavailable/corrupt storage
  src/examples.ts        one canned, hand-verified EXPLAIN dump per engine ("Try an example")
  src/engine.ts          Engine union type + label/validation helpers
  src/main.ts            wires the DOM: select/textarea/button -> parser -> tree-view, the
                          inspector, jump-to-hottest, example buttons, and storage restore/clear
  src/style.css           docs/DESIGN.md's swiss-grid modernist tokens and component styles
  src/wasm/               generated by scripts/build-wasm.sh - not committed
```

## Data flow

1. User pastes plan text, picks an engine, clicks Visualize (`main.ts`).
2. `main.ts` calls `ensureParserReady()` (`parser.ts`), which lazily `init()`s the wasm module.
3. `parsePlan(engine, text)` calls into wasm's `parse_plan`, which dispatches to the matching
   Rust parser (`lib.rs` -> `postgres.rs` / `mysql.rs` / `sqlite.rs`) and returns a `PlanNode`
   tree serialized via `serde-wasm-bindgen` (u64 fields arrive as plain JS numbers; absent
   `Option` fields arrive as `undefined`, not `null` - see `parser.test.ts`).
4. On success, `tree-view.ts` renders the tree: `findHottestNode` (self-time, not inclusive
   total time - see `PlanNode::self_time_ms` in `model.rs` and its TS mirror in `plan.ts`) marks
   the hotspot node, `isRowEstimateMismatch` flags >10x est-vs-actual divergence per node.
   Collapsing a node's toggle detaches its subtree from the DOM (not just visually hides it).
5. On failure (malformed input), the wasm side throws a bare string; `parser.ts` normalizes it
   into a real `Error`, and `main.ts` renders it as a styled inline error via `textContent`
   (never `innerHTML`, since error messages can echo back the user's pasted text).
6. `renderTree` returns a `TreeController` (`{element, focusPath, getCollapsedPaths}`), not a
   bare element: every node's DOM refs are keyed by its path (child-index sequence from root, see
   `pathKey` in `plan.ts`), so a collapsed ancestor chain can be re-expanded programmatically
   (`focusPath` - powers "Jump to hottest node", via `findHottestNodePath`) and expand state can
   be read back out for persistence (`getCollapsedPaths`).
7. Clicking a tree row (or Enter/Space while it's focused) calls `onNodeSelect`, which `main.ts`
   wires to `inspector.open()` (`inspector.ts`) - a focus-trapped dialog listing every non-null
   `PlanNode` field via `describeNodeFields`. Escape or a backdrop click closes it and returns
   focus to the row that opened it.
8. Every successful parse and every node toggle persists `{engine, text, collapsedPaths}` to
   `localStorage` (`storage.ts`, key `planscope:last-plan`); mounting the app restores it with
   no network round-trip. An explicit "Clear saved plan" button removes the entry. `examples.ts`
   holds one canned EXPLAIN dump per engine for the "Try an example" buttons, wired through the
   same `visualize()` path as a manual paste.

## Build

- **Parser**: `cargo test` (native) for fast iteration; `cargo build --target wasm32-unknown-unknown
  --release -p planscope-parser` for the actual artifact. `wasm-bindgen` (pinned to the same
  version as the crate dependency - see `crates/parser/Cargo.toml`) generates the JS/TS bindings
  the web app imports directly; a version mismatch between the two is a hard error, not a warning.
- **Web**: `scripts/build-wasm.sh` runs as an npm `pre*` hook (`predev`/`pretest`/`prebuild` in
  `web/package.json`) so `web/src/wasm/` always exists before anything that imports it runs.
  `vite.config.ts` sets `base: "./"` and emits into `site/`, so the built app works when hosted
  under a subpath (`apps.charliekrug.com/planscope`), not just at a domain root.
- **Production site**: `scripts/build-site.sh` invokes the same Vite/WASM build and leaves the
  complete static application in `site/`. The portfolio URL opens the visualizer directly;
  product guidance and search-focused copy live below the interactive workspace.
- **CI** (`.github/workflows/ci.yml`): a `parser` job runs `cargo fmt`/`clippy`/`test`/wasm build;
  a `web` job installs a matching Rust toolchain + `wasm-bindgen-cli` (via `jetli/wasm-bindgen-action`)
  before `npm ci`/`lint`/`test`/`build`, since the web build depends on the generated bindings.

## Testing approach

- Rust: unit tests per parser module (happy path + malformed input) plus `tests/basic.rs` for
  cross-engine smoke coverage and the `self_time_ms` behavior.
- TypeScript: `plan.test.ts` (pure helpers), `parser.test.ts` (mocks `./wasm/planscope_parser.js`
  via `vi.mock` so tests don't depend on a real wasm module), `tree-view.test.ts` (jsdom, checks
  depth/collapse/hotspot/badge/path/focus behavior on real DOM nodes), `inspector.test.ts` (open/
  close lifecycle, Escape, backdrop-vs-panel clicks), `storage.test.ts` (round-trip, clear,
  corrupt-JSON handling), `examples.test.ts` (loads the real compiled wasm via `initSync` + raw
  bytes read from disk - the one place tests intentionally skip the mock, so a parser regression
  or a stale fixture fails here instead of only in the browser), `main.test.ts` (mocks `./parser`
  to test the UI wiring - empty input, success/error paths, inspector/jump/example/storage
  wiring - without touching wasm at all).
- Coverage: `cargo llvm-cov --workspace --summary-only` measures the Rust parser; `npm run
  test:coverage` measures the TypeScript UI with V8 coverage.
